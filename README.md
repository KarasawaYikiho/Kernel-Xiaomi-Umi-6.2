# Kernel-Xiaomi-SM8250

English | [简体中文](README-Zh-Cn.md)

Kernel porting orchestrator for Xiaomi SM8250-family devices. CI-driven migration from SO-TS 4.19 to a 5+ baseline, with `umi` retained as the current reference baseline.

> This repo is **not** a kernel source tree.

## Scope

- Phase2 migration automation via GitHub Actions
- Structured diagnostics and metrics
- Flash-ready artifact delivery

## Quick Start

Run **`ROM-Aligned-Umi-Port.yml`** workflow for the current reference device baseline, then inspect:

1. `artifacts/phase2-report.txt`
2. `artifacts/next-focus.txt`
3. `artifacts/runtime-validation-summary.md`
4. `artifacts/anykernel-info.txt`

## Inputs

| Input | Default | Description |
|-------|---------|-------------|
| `device` | `umi` | Device codename; current default baseline remains `umi` |

Source locations, default branches, toolchain source, and boot image baseline paths are no longer workflow inputs. The workflow auto-fills them in code:

- Source repo, target repo, default branches, toolchain download source, and ROM defaults are centralized in `Porting/Sm8250PortConfig.json`
- Local ROM dir for maintainer runs: `D:\GIT\MIUI_UMI`
- Local ROM zip fallback for maintainer runs: `D:\GIT\MIUI_UMI_OS1.0.5.0.TJBCNXM_d01651ed86_13.0.zip`
- GitHub Action fallback: tracked `Porting/OfficialRomBaseline/BootImgParts/` chunks auto-reassembled in CI
- `BOOTIMG_REQUIRED_BYTES`: derived from the active ROM baseline first, then falls back only when baseline metadata is unavailable

## Workflows

- **`ROM-Aligned-Umi-Port.yml`** — Main ROM-aware migration workflow for the current SM8250 baseline
- **`Build-Umi-Kernel.yml`** — Reference cloud build for the current SM8250 baseline

## Local Boot Baseline

For the shortest Windows path on this machine, run:

Use `Tools/Porting/RunLocalOfficialRomBaseline.ps1`; refreshed outputs and local validation details stay centralized in `Tools/Porting/README.md` to avoid duplicated maintenance.

It defaults to `D:\GIT\MIUI_UMI` and keeps the official `boot.img` out of git.

## Repository Hygiene

- `artifacts/`, `source/`, and `target/` are local working directories generated by workflows and validation runs
- `source.extract/`, `target.extract/`, and `source.zip` are local fetch/extract outputs and should also stay untracked
- `.ruff_cache/` and `__pycache__/` are local tool caches
- Keep those paths untracked so the repo stays reviewable and clean
- `Porting/OfficialRomBaseline/` stores ROM baseline metadata plus compact validation images intended for repo-side checks
- See `Porting-Plan.md` for the execution baseline, `Porting/README.md` for document order, and `Tools/Porting/README.md` for validation entry points
